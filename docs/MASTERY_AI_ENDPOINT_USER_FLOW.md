# Mastery AI Endpoint User Story and Integration Guide

Last updated: February 27, 2026  
Project: Mastery AI (Curriculum-Bound, Teacher-Assisted Personalized AI Tutor)  
Audience: Frontend team, AI team, backend team, product team, non-technical reviewers

## 1) Why this document exists

This guide explains, in plain English:

1. How a student uses Mastery AI from sign-up to quiz feedback.
2. Which endpoint is called at each step.
3. What payload each endpoint expects.
4. What response the frontend should store and pass to the next endpoint.
5. How backend and internal services communicate behind the scenes.

This is written as a user story with real names so every team member can follow the same flow without guessing.

## 2) Story characters

1. **Olasquare**: Student (SSS2, Term 1; studying Math and English).
2. **Adebimpe**: Teacher (checks class performance later).
3. **Mary**: Admin/content manager (curriculum governance stage).

## 3) Project purpose in plain English

Mastery AI is not an open chatbot.  
It is a guided tutor for SSS1-SSS3 students in Math, English, and Civic.  
The system uses:

1. Student profile and study history.
2. Curriculum and topic scope (SSS level + term + subject).
3. Diagnostic and quiz performance.
4. Graph-based concept mastery updates.

Goal: give each student the right next lesson, quiz, and feedback based on their actual learning gaps.

## 4) Core integration rules (important)

1. **Use JWT after login** for protected endpoints (`Authorization: Bearer <token>`).
2. Token contains `user_id` and `student_id` claims (same UUID value).
3. `student_id` in payload must match authenticated user for protected student endpoints.
4. `quiz_id` is generated by backend from `POST /learning/quizzes/generate`; frontend must store it and reuse it.
5. `attempt_id` comes from `POST /learning/quizzes/{quiz_id}/submit`; frontend must store it for results page.
6. Internal endpoints under `/api/v1/internal/*` are for service-to-service calls, not direct frontend use.

## 5) End-to-end flow in dependency order

This is the exact recommended order for frontend and QA.

1. Load metadata dropdowns:
   - `GET /api/v1/metadata/subjects`
   - `GET /api/v1/metadata/levels`
2. Register + login:
   - `POST /api/v1/auth/register`
   - `POST /api/v1/auth/login`
3. Setup student context:
   - `GET /api/v1/users/me`
   - `PUT /api/v1/users/me`
   - `POST /api/v1/students/profile/setup`
   - `GET /api/v1/students/profile`
   - `PUT /api/v1/users/{user_id}/preferences`
4. Start learning content:
   - `GET /api/v1/learning/topics`
   - `GET /api/v1/learning/topics/{topic_id}/lesson`
5. Track activity + dashboard cards:
   - `POST /api/v1/learning/activity/log`
   - `GET /api/v1/students/stats`
   - `GET /api/v1/students/leaderboard`
6. Tutor session lifecycle:
   - `POST /api/v1/tutor/sessions/start`
   - `GET /api/v1/tutor/sessions/{session_id}/history`
   - `POST /api/v1/tutor/sessions/{session_id}/end`
7. Diagnostic and adaptive path:
   - `POST /api/v1/learning/diagnostic/start`
   - `POST /api/v1/learning/diagnostic/submit`
   - `POST /api/v1/learning/path/next`
   - `GET /api/v1/learning/path/map/visual`
8. Quiz cycle:
   - `POST /api/v1/learning/quizzes/generate`
   - `POST /api/v1/learning/quizzes/{quiz_id}/submit`
   - `GET /api/v1/learning/quizzes/{quiz_id}/results`

## 6) Detailed endpoint guide (public endpoints)

### 6.1 Metadata endpoints (no auth)

#### GET /api/v1/metadata/subjects

Who calls it: Frontend at app startup and onboarding pages.  
Purpose: populate subject dropdown.

Response example:

```json
{
  "subjects": ["math", "english", "civic"]
}
```

Frontend action:
1. Render subjects in selector.
2. Keep returned values as canonical payload values.

#### GET /api/v1/metadata/levels

Purpose: populate level/term selectors.

Response example:

```json
{
  "levels": ["SSS1", "SSS2", "SSS3"],
  "terms": [1, 2, 3]
}
```

Frontend action:
1. Use these exact level and term values in profile setup and later flows.

### 6.2 Auth endpoints

Story step: Olasquare creates an account.

Request:

```json
{
  "email": "olasquare@example.com",
  "password": "StrongPass123",
  "role": "student",
  "first_name": "Olasquare",
  "last_name": "Adeola"
}
```

Success response:

```json
{
  "user_id": "27ae55f0-0736-4d3c-9ebb-b8e30bde8ac5",
  "email": "olasquare@example.com",
  "role": "student",
  "first_name": "Olasquare",
  "last_name": "Adeola",
  "display_name": "Olasquare Adeola",
  "message": "User created successfully."
}
```

Frontend action:
1. Optionally auto-navigate to login.
2. Store `user_id` only for immediate flow convenience (login still required for token).

#### POST /api/v1/auth/login

Story step: Olasquare signs in.

Request:

```json
{
  "email": "olasquare@example.com",
  "password": "StrongPass123"
}
```

Success response:

```json
{
  "access_token": "<JWT_TOKEN>",
  "user_id": "27ae55f0-0736-4d3c-9ebb-b8e30bde8ac5",
  "role": "student",
  "first_name": "Olasquare",
  "last_name": "Adeola",
  "display_name": "Olasquare Adeola"
}
```

Frontend action:
1. Store token securely.
2. Attach `Authorization: Bearer <token>` to protected requests.
3. Store `user_id` as current student identifier.
4. If frontend decodes token, it should expect:
   - `sub` (email),
   - `role`,
   - `user_id`,
   - `student_id` (same UUID as `user_id`).

#### PUT /api/v1/auth/password

Story step: Olasquare changes password from settings page.

Headers:
1. `Authorization: Bearer <token>`

Request:

```json
{
  "current_password": "StrongPass123",
  "new_password": "NewStrongPass456"
}
```

Success response:

```json
{
  "message": "Password changed successfully."
}
```

## 7) Internal endpoints (service-to-service, not frontend)

These routes are for backend/ai-core/graph contracts.

### 7.1 Internal Postgres contracts

1. `GET /api/v1/internal/postgres/profile?student_id=...`
2. `GET /api/v1/internal/postgres/history?student_id=...&session_id=...`
3. `POST /api/v1/internal/postgres/quiz-attempt`
4. `GET /api/v1/internal/postgres/class-roster?class_id=...`

Why this matters:
1. AI-core can fetch student context/history without touching frontend.
2. Teams can integrate independently with stable contracts.

### 7.2 Internal Graph contracts

1. `GET /api/v1/internal/graph/context?...`
2. `POST /api/v1/internal/graph/update-mastery`

Why this matters:
1. Keeps concept mastery logic centralized.
2. Quiz and diagnostic flows can update mastery consistently.

## 8) Functional communication flow (high level)

### 8.1 Student onboarding flow

1. Frontend -> `/auth/register` -> user created.
2. Frontend -> `/auth/login` -> token + `user_id`.
3. Frontend -> `/users/me` -> personal identity card for dashboard.
4. Frontend -> `/students/profile/setup` -> scoped learning profile.
5. Frontend -> `/users/{user_id}/preferences` -> personalization saved.

### 8.2 Learning flow

1. Frontend -> `/learning/topics` -> topic list.
2. Frontend -> `/learning/topics/{topic_id}/lesson` -> lesson blocks.
3. Frontend -> `/learning/activity/log` -> points/streak tracking.

### 8.3 Quiz flow with backend orchestration

1. Frontend -> `/learning/quizzes/generate`.
2. Backend -> AI-core (question generation contract).
3. Backend stores quiz/questions and returns `quiz_id`.
4. Frontend -> `/learning/quizzes/{quiz_id}/submit`.
5. Backend grades + stores attempt + logs activity.
6. Backend -> `/internal/graph/update-mastery` (concept mastery update).
7. Frontend -> `/learning/quizzes/{quiz_id}/results`.
8. Backend optionally pulls insight text from AI-core and returns full result.

## 9) Common mistakes and how to avoid them

1. **Mistake**: Frontend uses email as `student_id`.  
   **Fix**: Always use UUID `user_id` from login response.

2. **Mistake**: Frontend invents `quiz_id` or `question_id`.  
   **Fix**: Use IDs returned by `/learning/quizzes/generate`.

3. **Mistake**: Missing Bearer token on protected endpoints.  
   **Fix**: Always attach `Authorization` header after login.

4. **Mistake**: Calling internal endpoints from frontend.  
   **Fix**: Frontend should only use public `/api/v1/...` endpoints designed for UI.

5. **Mistake**: Calling submit before generate.  
   **Fix**: Respect dependency chain: Generate -> Submit -> Results.

## 10) Quick frontend checklist

Before release, frontend should verify:

1. Token is stored and sent in headers where required.
2. `user_id` from login is used as `student_id` in payloads.
3. `quiz_id` and `attempt_id` are preserved across quiz screens.
4. Error messages from API are surfaced cleanly in UI.
5. Route order follows dependency flow in Section 5.

## 11) Final note for cross-team communication

If any endpoint contract changes:

1. Update OpenAPI schema.
2. Update this guide.
3. Announce in team channel with:
   - old payload,
   - new payload,
   - migration date,
   - impact on frontend/ai-core.

This avoids integration drift in a shared database and parallel team setup.
